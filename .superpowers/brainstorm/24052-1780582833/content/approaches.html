Three approaches to the FullCalendar migration

How tightly should FullCalendar own the UI? Choose how much control we hand over.

A

FC as view layer only — our toolbar stays (Recommended)

Replace only CalendarGrid and its sub-components with a FullCalendar wrapper. The custom CalendarToolbar (Today / prev / next / Week–Month toggle), CalendarPagination, and URL state are untouched. PlanCalendarPage passes the current date range to FC via an imperative ref call. FC fires eventDrop → mutation and dateClick → quick-create.

Pros

  • Smallest blast radius — plans 1 & 2 untouched
  • URL state stays as authoritative source of truth
  • Our toolbar design matches the rest of the app
  • Pagination logic stays unchanged

Cons

  • Need to keep FC in sync via imperative calendarRef API
  • FC's own date-navigation must be disabled (headerToolbar={false})
B

FC owns navigation — replace toolbar too

Remove our custom CalendarToolbar and CalendarPagination. Let FullCalendar render its own header. Override FC's CSS classes to match our design tokens. FC's datesSet callback updates the URL. FC's viewDidMount writes view mode to URL.

Pros

  • Less glue code for navigation sync
  • FC's toolbar gets keyboard nav / a11y for free

Cons

  • Harder to match our exact toolbar design
  • FC toolbar CSS overrides fight lint:styles rule
  • Pagination now fights FC's own event rendering model
  • Browser back/forward can desync FC vs URL
C

Approach A + Day view

Same as A (FC as view layer, our toolbar controls navigation), but also add a third view mode — timeGridDay — showing a single-day time-slot grid. Adds a "Day" button to the toolbar alongside Week / Month.

Pros

  • FullCalendar makes this nearly free to add
  • Richer experience for dense-task days

Cons

  • Wider scope — tasks need start/end times for time-grid to be useful
  • Our tasks mostly have date-level precision, not time slots
  • Backend + URL state need a new calDay param

My recommendation: A. It replaces exactly what needs replacing, preserves the URL-centric state model we built in plans 1–2, and keeps the blast radius small. Drag-and-drop works the same in all three options via FC's eventDrop.
